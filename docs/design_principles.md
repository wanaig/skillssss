## 一、核心原则

### 1.1 Harness Engineering 的本质

不是让 AI 一次做完所有事，而是设计一套"套件(harness)"——决定存储什么、检索什么、展示给模型什么。主智能体就是这个套件的编排者。

**核心矛盾**：上下文窗口有限 vs 任务复杂度无限。解法不是更大的窗口，而是更聪明的上下文管理。

### 1.2 三条铁律

| 铁律 | 含义 | 违反后果 |
|------|------|----------|
| **文件即记忆** | 子智能体的所有产出必须持久化到文件，不依赖内存 | 上下文丢失后无法恢复 |
| **隔离即常态** | 每个子智能体只看到主智能体给它的信息，不假设它知道外部状态 | 隐式依赖导致幻觉 |
| **记录即保险** | 主智能体和子智能体都写日志，确保一切可追溯 | 出问题后无法复盘 |

## 二、角色体系设计

### 2.1 三角色模型

```
┌─────────────┐
│   主智能体    │  编排者：拆任务、管状态、中转信息
│  (Orchestrator) │
└──────┬──────┘
       │
   ┌───┴───────────────┐
   │                   │
   ▼                   ▼
┌──────────┐     ┌──────────┐
│ 开发智能体 │     │ 测试智能体 │
│  (Dev)   │     │ (Tester)  │
└──────────┘     └──────────┘
```

### 2.2 各角色核心职责

| 角色 | 职责 | 工具权限 | 上下文管理 |
|------|------|----------|------------|
| **主智能体** | 拆任务、管计划、中转信息、写日志 | 全部 | 保持全程上下文 |
| **开发智能体** | 读需求→写代码→自测→输出 | 读写 + Bash | 每个任务新创建或 resume |
| **测试智能体** | 读需求→审查代码→输出结论 | 只读 + Bash | 每个任务新创建或 resume |

### 2.3 开发智能体：执行者

**核心原则**：每个子任务创建一个新子智能体，或 resume 上一个同任务的开发智能体。

为什么不用一个长对话的开发智能体？
- 上下文窗口会被历史占满
- 长对话后模型注意力分散
- 出错后难以定位问题

为什么 resume 而不是新建？
- 测试智能体的反馈需要结合开发时的上下文才能理解
- resume 让开发智能体能"看到"自己之前写了什么

### 2.4 测试智能体：质检员

**核心原则**：只读，不修改任何代码文件。

**测试维度的设计原则**：
- **可量化**：每项检查有明确的通过/失败标准
- **分优先级**：blocker/major/minor，明确什么算通过、什么算失败

## 三、上下文管理策略

### 3.1 核心矛盾

| 方案 | 优点 | 缺点 |
|------|------|------|
| **一个长对话** | 上下文完整 | 窗口爆满，注意力分散 |
| **每任务新智能体** | 上下文干净 | 丢失历史，重复阅读 |
| **新智能体 + 文件传递** | 干净且有记录 | 需要精心设计传入内容 |

**推荐方案**：新智能体 + 精心设计的文件传递 + 经验库积累。

### 3.2 每个子智能体必须读取的上下文

**开发智能体的必读清单**：

1. 任务描述 → 知道做什么
2. 需求文档（对应 section）→ 知道做成什么样
3. 开发规范文件 → 知道怎么写
4. 已完成的代码结构 → 知道已有的模式
5. 经验教训库（lessons-learned.md）→ 知道踩过的坑
6. 开发计划（dev-plan.md）→ 知道在全局中的位置

**测试智能体的必读清单**：

1. 任务描述 → 知道测什么
2. 需求文档 → 知道验收标准
3. 待测代码文件 → 审查对象
4. 开发规范 → 知道合规标准
5. 主题/样式配置 → 视觉基准

### 3.3 经验库设计

经验库是跨子智能体传递知识的唯一可靠通道。由开发Agent 在修正后更新。

**经验总结的粒度原则**：
- 太细 ×："Page03 第 42 行的 left 值应该是 120 而不是 100"
- 太宽 ×："注意布局要好看"
- 恰好 ✓："标题和正文字号差距至少 8px"

**三条原则**：
1. **原则性 > 数值性**：写"为什么错"而非"改了什么值"
2. **模式级 > 页面级**：写"哪种业务场景容易犯这个错"
3. **可迁移 > 可复制**：下个项目还能用吗？

## 四、开发计划拆分策略

### 4.1 拆分粒度

| 太粗 | 恰好 | 太细 |
|------|------|------|
| "做完整套动画" | "开发 Page03 - 数据图表页" | "写第 12 行的 interpolate" |
| > 500 行代码 | 100-300 行代码 | < 30 行代码 |

**黄金法则**：一个子任务 = 一个可独立运行和验证的功能单元。

### 4.2 拆分顺序

```
Phase 0: 框架搭建
  → 项目初始化、公共组件、主题配置、路由配置
  → 由 Planner Agent 完成（不需要开发Agent）

Phase 1~N: 逐功能开发
  → 每个功能一个子任务
  → 每个子任务走完整的开发-测试循环

Phase Last: 整体集成测试
  → 所有功能完成后，做一次整体测试
```

### 4.3 依赖关系处理

- 如果任务 B 依赖任务 A 的产出，A 必须先完成并通过测试
- 公共组件应该在 Phase 0 就完成
- 如果发现新依赖，暂停当前任务，先完成依赖项

## 五、日志与记录体系

### 5.1 三层日志架构

```
主智能体日志（全流程）
  │
  ├── 子智能体日志（各自独立）
  │     ├── 开发智能体日志
  │     └── 测试智能体日志
  │
  └── 经验库（跨任务积累）
```

### 5.2 数据访问边界

主智能体的"不读内容"原则有明确边界：

| 数据项 | 主Agent 是否可读 | 方式 |
|--------|-----------------|------|
| 架构/需求文档 | 否 | 只传路径 |
| dev-plan.md | 是 | 仅读任务列表 |
| test-report.json | 是 | 仅读 verdict/severity |
| 测试报告全文 | 否 | 路径传给开发Agent |
| 源代码 | 否 | 全部委托 |

## 六、常见问题

| 问题 | 原因 | 解法 |
|------|------|------|
| Agent(resume) 找不到 ID | 路径不对 | 使用 agent-registry/{key}.json 精确索引 |
| 开发智能体不理解测试反馈 | 传给新Agent 而非 resume | 修正循环必须 resume 原Agent |
| 测试智能体误判 | 检查标准不明确 | 增加验证步骤 |
| 经验库太长读不完 | 积累了太多条目 | 定期精简，只保留通用高价值的 |
| 开发智能体重复踩坑 | 没读经验库 | prompt 中强制要求先读 lessons-learned.md |

## 七、适用场景

**适用**：
- 大型开发任务，可拆分为独立子任务
- 需要反复迭代验证的任务
- 需要严格质量把控的项目

**不适用**：
- 简单的一次性任务
- 任务间强耦合、无法独立验证
- 不需要迭代修正的场景
