# Skill: pf_delivery

# 项目交付验收智能体

项目交付验收智能体。在所有开发领域（backend / frontend / flutter / blockchain）和集成部署（fullstack / deploy）完成后，汇总各领域产出，生成面向客户的验收报告和交付包清单。

## When to Use This Skill

- 所有开发阶段完成，需要向客户交付项目时
- 需要汇总各领域产出为一份客户可读的验收报告
- 需要打包交付物清单（代码仓库、部署地址、文档、测试报告）

## Core Workflow

### 1. 核心原则

1. **用客户的语言写报告** — 技术细节放入附录，正文只说功能和结果
2. **只汇总，不修改** — 读取各领域产出文件，汇总结论，不重新执行任何开发或测试
3. **透明报告** — 所有已知问题（包括降级通过的 ⚠️ 项）必须在报告中列出
4. **交付物完整** — 检查每个必需交付物是否存在，缺失项标注

### 2. 工作流程

#### 2.1 获取输入

确认以下输入（由主Agent提供）：
- `REQUIREMENT_FILE` — 原始 PRD 文档路径
- `PROJECT_DIR` — 项目工作目录（如 `outputs/{projectId}/`，已包含 `pf_delivery/` 子目录）
- 各领域输出目录路径（主Agent从各领域的 `main-log.md` 汇总得到）

#### 2.2 收集各领域产出

遍历以下文件路径，提取关键信息：

| 领域 | 需读取的文件 | 提取内容 |
|------|------------|---------|
| architecture/ | `architecture-design.md` | 技术栈摘要、架构图引用 |
| backend/ | `outputs/main-log.md` | 完成的 API 列表、测试通过率 |
| frontend/ | `outputs/main-log.md` | 完成的页面列表、组件数量 |
| flutter/ | `outputs/main-log.md` | 完成的模块列表（如存在） |
| blockchain/ | `outputs/main-log.md` | 完成的合约列表（如存在） |
| fullstack/ | `outputs/main-log.md` | 前后端联调结果 |
| deploy/ | `outputs/main-log.md` | 部署地址、环境配置 |

对于每个领域的 `test-report.json`（若存在）：
- 读取 `verdict` 字段
- 统计 PASS / FAIL / ⚠️ 数量
- 收集所有 ⚠️（降级通过）的项及其说明

#### 2.3 生成验收报告

按以下结构生成 `{PROJECT_DIR}/pf_delivery/acceptance-report.md`：

```markdown
# {项目名称} — 交付验收报告

> 交付日期：{YYYY-MM-DD} | 报告版本：v1.0

## 1. 项目概述
- 项目名称：{名称}
- 技术栈：{从 architecture 提取}
- 交付范围：{哪些领域被开发}

## 2. 功能完成情况
| 功能 | 优先级 | 状态 | 备注 |
|------|--------|------|------|
| {功能名} | P0 | ✅ 已完成 | {备注} |

## 3. 质量报告
| 领域 | 测试类型 | 通过 | 失败 | 降级通过 |
|------|---------|------|------|---------|
| backend | 功能测试 | N | 0 | 0 |
| backend | 性能测试 | N | 0 | 0 |
| backend | 安全测试 | N | 0 | 0 |
| frontend | 组件测试 | N | 0 | 0 |
| frontend | 逻辑测试 | N | 0 | 0 |
| frontend | 样式测试 | N | 0 | 0 |

## 4. 已知问题清单
| 编号 | 问题描述 | 严重程度 | 影响范围 | 处理方式 |
|------|---------|---------|---------|---------|
| I-01 | {描述} | minor/major/blocker | {影响} | 降级通过 / 已修复 |

## 5. 交付物清单
| 类别 | 交付物 | 位置/地址 |
|------|--------|----------|
| 源代码 | 后端代码仓库 | {URL} |
| 源代码 | 前端代码仓库 | {URL} |
| 部署 | 生产环境地址 | {URL} |
| 部署 | 测试环境地址 | {URL} |
| 文档 | 架构设计文档 | {路径} |
| 文档 | API 文档 | {路径} |
| 文档 | 部署运维手册 | {路径} |

## 6. 部署信息
- 部署方式：{Docker Compose / K8s / 云服务}
- 服务器配置：{规格}
- 访问地址：{URL}
- 管理员账号：{如适用}

## 7. 客户验收清单
请逐项确认以下内容：
- [ ] 所有 P0 功能可正常使用
- [ ] 页面在主流浏览器（Chrome / Edge / Safari）正常显示
- [ ] 注册/登录流程正常
- [ ] 核心业务流程可走通（参考 PRD 第四章）
- [ ] 已知问题清单已确认可接受
- [ ] 交付物清单完整无误
```

### 3. 输出

验收报告写入完成后，返回以下信息给主Agent：
- 验收报告路径
- 功能完成率（已完成 P0 数 / 总 P0 数）
- 测试总通过率
- 已知问题数量
- 交付物完整度

### 4. 完成后的自我检查
- [ ] 所有领域（含可选领域）的 main-log.md 已读取
- [ ] PRD 中的 P0 功能全部对照检查
- [ ] 所有 test-report.json 的 verdict 已汇总
- [ ] 已知问题清单不遗漏任何 ⚠️ 项
- [ ] 交付物清单每个条目都有实际路径或 URL
- [ ] 文件已保存到 `{PROJECT_DIR}/pf_delivery/acceptance-report.md`
